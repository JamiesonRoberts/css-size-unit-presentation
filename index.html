And here is where things get really skrewed up

An inch doens't always equal an inch:

Pixels are dependant on the size of the "inch"

or an "inch" are relative to a pixel.

Lo-Res Screens

The unit "px" is reference to a physical pixel

so an inch is 96 reference pixels in size

and all other absolute length units adjust accordingly

Hi-Res Screens or Physical item

The "inch" is defined by its physical size, so...

an inch is an inch

a pixel is 1 / 96th of an inch

a point is 1 / 72th of an inch

and a pica is 1 / 6th of an inch

Just like the APS or DPS

And this is just 1 type of unit size

of 6.

Don't worry, most are digital constructions.

Relative Lengths

There are 2 types:

Font-relative Lengths

Viewport-percentage Lengths

Font-relative Lengths

EM, REM*, CH, EX

All except for REM's are relative to the font metric of the element they are applied to

unit relative to
em font size of the element
ex x-height of the element’s font
ch width of the "0" (ZERO, U+0030) glyph in the element’s font
rem font size of the root element
give examples of each one

Viewport-relative Lengths

VH, VW, VMIN, VMAX

unit relative to
vw 1% of viewport’s width
vh 1% viewport’s height
vmin 1% of viewport’s smaller dimension
vmax 1% of viewport’s larger dimension
vw unit Equal to 1% of the width of the initial containing block. In the example below, if the width of the viewport is 200mm, the font size of h1 elements will be 16mm (i.e. (8×200mm)/100). h1 { font-size: 8vw } vh unit Equal to 1% of the height of the initial containing block. vmin unit Equal to the smaller of vw or vh. vmax unit Equal to the larger of vw or vh.

So now that we know all that

Can we all agree that these absolute units of size are actually pretty terrible for web. But great for print.

That aside: lets go over some examples

Where pixels and the like break down for web

@media all and (min-width: 768px)

≠ 

@media all and (min-width: 48em)

Let me explain why

@media all and (min-width: 768px)

Will still = true if you increase your browser magnification (read font size)

Because the actual number of pixels of the window haven't changed in relation to the font size increase

@media all and (min-width: 48em)

Won't.

Because the actual number of EM's of screen real estate across the screen won't evaluate to true if you increase the font size.

EM's vs REM's

Why you should care about the difference

EM's are relative to the font size of the element or the last parent element with a specified font size

As you increase the font size, all other EM measurements increase proportionally

Which is great when you want to maintain padding, borders, image sizes relative to a single element

REM's?
The red-headed step child

Are relational in size to the root elements size (ie HTML tag)

Meaning if you increase the font size of the text, the padding of the element will still be the same and its overall size will be as well... just with bigger letters in it.

It all comes down to inheritance

Each have a use case, depending on how you want things to inherit sizing

Or weird fonts.

Or better yet multiple fonts with different baseline sizes.

Viewport relative sizing

Really sweet units that gets no love

Only thing missing is IE8 support

But VH and VW are supported on IE9 forward
